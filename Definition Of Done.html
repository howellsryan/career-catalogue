Definition of Done

Conditions for a story to be considered complete within a sprint

Introduction

The Definition of Done represents a set of conditions for a story undertaken by the development team to be considered complete (in regards to a sprint). These criteria ensure quality and consistency across all delivered work.

Key Criteria

Testing Complete

Tests written and passing for all new and modified functionality.

CI Passing

Continuous Integration build is successfully passing.

Security Verified

Security vulnerability scan passing with no critical issues.

Peer Reviewed

Code has been peer-reviewed by at least one other team member.

Documentation

Documentation has been updated to reflect changes.

Acceptance Criteria

We believe all acceptance criteria defined in the story are met.

Production Ready

Ready for Production!

The change is considered deliverable and Production ready. When possible, it should be deployed to Production as part of being "Done".

Important Note

A story is not considered "Done" until it has been deployed to Production (where possible and scenario-appropriate).

QA Involvement

Quality Assurance is a critical component of our Definition of Done process. QA should be involved throughout development rather than just at the end.

  1. Early Involvement: QA should be included in story refinement to understand requirements.
  2. Ongoing Testing: Testing should happen throughout development, not just at the end.
  3. Final Verification: QA performs final verification against acceptance criteria.
  4. Sign-off Required: If there are issues found at the last stage, the item cannot be marked as done.
